# 三版本迭代计划（V1.0 / V2.0 / V3.0）

> **项目**: AI-driven 城市智慧停车管理与诱导系统
> **PM**: 丁梓钊 | **日期**: 2026-05-29
>
> 基于 PRD V0.2 评审结果制定，明确三个版本的交付范围、里程碑和验收标准。

---

|------|----------|------|:------:|:----:|

## 二、V1.0 — 核心闭环 MVP（第4-6周）

### 2.1 版本目标

> 实现车主端全流程闭环 + 管理端基础运营功能，达到可演示级别。

### 2.2 交付范围

#### 车主端（6模块）

| 模块 | 功能点 | 负责人 | 工时 |
|------|--------|:------:|:----:|
| M1 注册登录 | 手机号注册、登录、Session管理 | 程晓洋 | 0.5d |
| M2 车场浏览 | 车场列表(分页+搜索)、车场详情 | 程晓洋 | 1.0d |
| M3 车位查询 | 车位列表、实时状态筛选、按距离/价格排序 | 程子浩 | 1.0d |
| M4 车位预约 | 选车位→锁定15分钟→到场确认/取消、超时释放、并发控制 | 程子浩 | 1.5d |
| M5 计费支付 | 入场计时→前15分钟免费→出场计费→模拟支付→订单生成 | 程子浩 | 1.0d |
| M6 记录寻车 | 停车历史记录、当前车位查询、纯文字步行指引 | 丁梓钊 | 0.5d |
| M7 车主首页 | 搜索入口、快捷入口、进行中预约/订单展示 | 丁梓钊 | 1.0d |

#### 管理端（2模块）

| 模块 | 功能点 | 负责人 | 工时 |
|------|--------|:------:|:----:|
| M8 运营后台 | 车场CRUD、车位状态一览、收入/车流统计 | 程晓洋 | 1.0d |
| M9 数据看板 | ECharts × 4图表（周转率/高峰/收入/利用率） | 程晓洋 | 1.0d |

#### 基础设施

| 项 | 内容 | 负责人 |
|----|------|:------:|
| 数据库建表 | 6张表DDL + SQLAlchemy模型 + ER图 | 程晓洋 |
| 项目脚手架 | Vue3 + FastAPI + SQLite + 目录结构 | 程子浩 |
| API规范 | Swagger自动文档、统一返回格式 `{code, msg, data}` | 程子浩 |
| 测试数据 | 50车场+500车位+5管理员+20车主模拟数据 | 丁梓钊 |
| 公共组件 | 导航栏、车场卡片、车位状态标签、倒计时组件 | 程晓洋 |

### 2.3 里程碑

```
Week 4 (6.15-6.21) ─── 基础设施+数据库+M1+M2+M3
Week 5 (6.22-6.28) ─── M4+M5+M6+M7 核心流程联调
Week 6 (6.29-7.5)  ─── M8+M9 管理端+全局联调+Bug修复
```

### 2.4 V1.0 验收标准

| # | 标准 | 验证方式 |
|:--:|------|----------|
| 1 | 用户可完成注册→登录→搜索车场→查看车位→预约→支付→寻车全流程 | E2E流程演示 |
| 2 | 预约状态机正确：free→locked(15min)→occupied→free | 边界测试 |
| 3 | 计费规则正确：≤15min免费、>15min按整时计 | 边界测试 |
| 4 | 并发预约正确：同一车位只能被一个用户锁定 | 并发测试 |
| 5 | 数据看板展示4张ECharts图表且数据正确 | 数据校验 |
| 6 | API响应时间 ≤ 500ms，页面加载 ≤ 2s | 性能测试 |
| 7 | Chrome/Edge均可正常运行 | 兼容性测试 |

## 四、V3.0 — 打磨完善（第8周）

### 4.1 版本目标

> 演示质量打磨 + 文档完善 + 答辩演练，确保最终验收展示效果。

### 4.2 交付范围

| 项 | 内容 | 负责人 |
|----|------|:------:|
| 演示脚本 | 编写10分钟演示流程（覆盖全部核心功能） | 全员 |
| 演示数据 | 构造"有故事感"的演示数据（真实车牌号、合理时间线） | 丁梓钊 |
| UI最终打磨 | 统一色彩/字体/间距、过渡动画、Loading状态 | 丁梓钊 |
| 性能优化 | API响应优化、前端懒加载、图表按需渲染 | 程子浩 |
| 用户手册 | 面向答辩评委的操作手册（截图+说明） | 程晓洋 |
| 项目总结报告 | 实训个人总结合并（3人各一份） | 全员 |
| 答辩排练 | 至少完整走2遍演示流程 | 全员 |

### 4.3 里程碑

```
Week 8 (7.6-7.12)   ─── 演示准备 + 文档输出
Week 8 (7.13-7.18)  ─── 答辩排练 + 最终提交
```

### 4.4 V3.0 验收标准

| # | 标准 |
|:--:|------|
| 1 | 演示流程不超过12分钟，覆盖注册→预约→支付→寻车→看板全链路 |
| 2 | UI视觉统一、无明显粗糙感 |
| 3 | 无演示中崩溃或明显Bug |
| 4 | 全部文档（用户手册+总结报告+答辩PPT）已提交 |
| 5 | 团队至少完成2次完整答辩排练 |

## 六、风险管理

| 风险 | 版本阶段 | 规避方案 |
|------|:--------:|----------|
| V1.0开发延期 | V1.0→V2.0 | M10/M11为P2，延迟不影响核心演示 |
| 联调Bug过多 | V1.0 | 第6周专设Bug修复期，M10/M11可砍 |
| 队友进度落后 | V1.0 | 模块独立解耦，组长兜底M1/M2/M8/M9 |
| 演示数据不真实 | V3.0 | 提前准备合理车牌号+时间线，构造真实感 |
| 答辩网络问题 | V3.0 | 全本地运行，零外部依赖 |
