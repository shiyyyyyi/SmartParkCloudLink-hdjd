## 一、需求采集

### 1.1 用户调研

| 用户类型 | 核心需求 | 痛点 |
|------|------|------|
| 私家车主 | 快速找到空余车位、提前预约、无感支付 | 盲目兜圈找位耗时、不同停车场不同APP、支付繁琐 |
| 停车场管理员 | 管理车位状态、查看收入统计、处理异常 | 人工记录易出错、无法实时掌握车位情况、收入统计靠Excel |
| 车位主（共享） | 闲置时段出租变现 | 无发布渠道、担心超时占用 |
| 城市交管部门 | 掌握全市停车态势、制定停车政策 | 各车场数据孤岛、无法全局分析 |

### 1.2 核心场景

```
场景1：车主出行前 → 搜索目的地周边停车场 → 查看实时空余车位 → 预约锁定 → 导航前往
场景2：车主到达 → 入场计时开始 → 出场时自动计费 → 模拟支付完成 → 查看停车记录
场景3：车主返回取车 → 输入车牌号 → 系统显示车位位置 → 步行导航指引找车
场景4：车场管理员 → 登录后台 → 查看实时车位占用 → 查看当日收入 → 导出统计报表
场景5：车位主 → 发布闲置时段 → 设置价格 → 车主预约使用 → 到期自动释放
```

## 二、竞品分析

### 2.1 主要竞品

| 产品 | 优势 | 不足 |
|------|------|------|
| **ETCP停车** | 覆盖广、车牌识别成熟 | 各车场独立运营，无跨车场统一查询 |
| **捷停车** | 车位预约功能完善 | 仅限合作车场，二三线城市覆盖弱 |
| **停简单** | 支付体验流畅 | 功能偏支付工具，缺少管理分析 |
| **各商场自有停车系统** | 与会员体系打通 | 一个商场一个APP，用户碎片化 |

### 2.2 差异化定位

| 维度 | 竞品 | 本项目 |
|------|:---:|:---:|
| 多车场统一查询 | ⚠️ 各自为政 | ✅ 一个平台查所有入驻车场 |
| 在线预约锁定 | ⚠️ 部分支持 | ✅ 选车位→锁定15分钟→到场 |
| 反向寻车 | ⚠️ 少数有 | ✅ 输入车牌→显示位置→步行引导 |
| 共享车位 | ❌ 极少 | ✅ C2C车位共享，闲置变现 |
| 运营数据看板 | ⚠️ 简单统计 | ✅ ECharts多维度可视化 |

## 三、技术分析

### 3.1 技术栈

| 层级 | 选型 | 版本 | 说明 |
|------|------|------|------|
| 前端 | Vue 3 + Element Plus + ECharts | 3.x | 组件化UI、数据可视化 |
| 后端 | FastAPI | 0.100+ | 异步高性能、Swagger自动文档 |
| 数据库 | SQLite + SQLAlchemy | 内置 | 零部署、轻量级 |
| 支付 | 模拟支付 | — | 不接入真实支付，前端模拟流程 |
| 地图 | 静态示意图/列表 | — | 不接入GIS，用停车场列表+简单示意图替代 |

### 3.2 技术可行性

| 关键点 | 判断 | 说明 |
|------|:---:|------|
| 多车场数据管理 | ✅ 可行 | SQLite关系型数据库，车场-车位-订单三表关联 |
| 车位预约锁定 | ✅ 可行 | 状态机设计（空闲→锁定→占用→释放），Redis不是必需的 |
| 计费逻辑 | ✅ 可行 | 按时长×费率，纯Python后端计算 |
| 数据可视化 | ✅ 可行 | ECharts直接渲染，后端查数据给前端即可 |
| 反向寻车 | ✅ 可行 | 简单实现：记录车位编号，用文字+示意图指引方向 |
| 三人并行开发 | ✅ 可行 | 前后端分离，模块独立，Git分支管理 |

### 4.2 优先级分级

| 优先级 | 模块 | 理由 |
|:---:|------|------|
| **P0 核心** | M1注册、M2车场、M3车位查询 | 系统基础，没这三样系统跑不起来 |
| **P0 核心** | M4预约、M5计费、M7车主端 | 核心业务流程闭环 |
| **P1 重要** | M6寻车、M8运营后台、M9看板 | 差异化亮点+管理价值 |
| **P2 增强** | M10共享、M11违停 | 锦上添花，时间不够可降级 |

### 4.3 核心功能用例

#### UC-01：车位查询与预约

| 项 | 内容 |
|------|------|
| 前置条件 | 用户已登录，系统中已录入停车场 |
| 触发 | 用户在车主端搜索目的地 |
| 流程 | ①输入目的地 ②展示周边车场列表（含空余车位数） ③选择车场查看车位详情 ④选择车位点击预约 ⑤锁定15分钟 ⑥到场确认 |
| 后置条件 | 车位状态变更为"已占用"，生成预约记录 |
| 异常 | 超时未确认→自动释放车位；车位已被他人锁定→提示重新选择 |

#### UC-02：停车计费与支付

| 项 | 内容 |
|------|------|
| 前置条件 | 车辆已入场，入场时间已记录 |
| 触发 | 车主点击"出场结算" |
| 流程 | ①计算停车时长 ②按费率计算费用 ③展示费用明细 ④模拟支付确认 ⑤生成订单记录 |
| 后置条件 | 车位释放为空闲，订单存入数据库 |
| 异常 | 支付失败→可重试，保留订单待支付状态 |

#### UC-03：反向寻车

| 项 | 内容 |
|------|------|
| 前置条件 | 用户有进行中的停车记录 |
| 触发 | 用户在车主端点击"找我的车" |
| 流程 | ①输入车牌号 ②查询当前占用车位 ③显示车位编号和所在区域 ④提供步行指引（文字+简单示意图） |
| 后置条件 | 无 |
| 异常 | 未找到记录→提示检查车牌号 |

## 五、可行性分析

### 5.1 技术可行性：✅ 高

- 全部为标准CRUD操作，无复杂算法依赖
- Vue3 + FastAPI + SQLite均为成熟技术栈
- 无外部API依赖，无联网风险，本地即可完整运行
- 三人均有Web开发基础

### 5.2 经济可行性：✅ 零成本

- 本地开发运行，无服务器费用
- 不接入真实支付，无第三方服务费
- 使用个人电脑，无设备支出

### 5.3 时间可行性：✅ 充裕

| 阶段 | 时间 | 产出 |
|------|------|------|
| 第1-2周 | 5.25-6.7 | 立项+计划+需求分析 |
| 第3周 | 6.8-6.14 | 系统设计+数据库建表+API骨架 |
| 第4-6周 | 6.15-7.5 | 11模块全量开发（9.5人天/3人≈3.2天） |
| 第7-8周 | 7.6-7.18 | 测试+文档+验收 |

**时间绰绰有余**，核心开发量仅9.5人天，3人并行不到一周即可完成。剩余时间可用于打磨UI、补充文档和答辩准备。

### 5.4 可行性结论

**项目完全可行**。技术门槛低、零成本、时间充裕，非常适合3人实训团队。最大优势是**零外部依赖**，答辩演示不受网络环境影响。
