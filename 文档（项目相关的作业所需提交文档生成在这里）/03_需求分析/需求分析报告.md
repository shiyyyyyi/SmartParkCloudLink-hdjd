# 需求分析报告 — AI-driven城市智慧停车管理与诱导系统

> 对应目录：03_需求分析 | 依据 `项目要求.txt` 的18个一级功能模块
> 课程：软件工程实训(A)（1521190120） | 第1-2周

---

## 一、需求采集

### 1.1 用户调研

| 用户类型 | 核心需求 | 痛点 |
|------|------|------|
| 私家车主 | 快速找到空余车位、提前预约、车牌识别入场、无感支付、反向寻车 | 盲目兜圈找位耗时、车场信息不透明、支付排队、取车困难 |
| 停车场管理员 | 管理车位状态、查看收入统计、处理设备故障、发布诱导信息 | 人工记录易错、设备状态分散、收入和流量统计滞后 |
| 车位主 | 闲置时段出租变现、查看预约和收益 | 缺少可信发布渠道、担心重复预约或超时占用 |
| 城市交管部门 | 掌握全市停车态势、监管违停、评估停车政策 | 路内外数据孤岛、缺少统一分析和监管视图 |
| 设备运维人员 | 监控摄像头、道闸、诱导屏、充电桩状态 | 故障发现慢、缺少统一台账和处理记录 |
| 商圈运营人员 | 配置停车优惠、核销停车券、评估活动效果 | 停车优惠与商户消费割裂，活动效果难统计 |

### 1.2 核心业务场景

```text
场景1：车主出行前 -> 搜索目的地周边停车场 -> 查看城市停车一张图 -> 预约锁定车位 -> 导航前往
场景2：车主到达停车场 -> 车牌识别入场 -> 订单开始计时 -> 出场自动计费 -> 模拟无感支付完成
场景3：车主返回取车 -> 输入车牌或车位号 -> 系统定位车位楼层区域 -> 显示步行寻车路线
场景4：停车场管理员 -> 查看运营后台 -> 维护车位/订单/收入/设备 -> 远程处理异常
场景5：交管部门 -> 查看监管大屏 -> 分析供需、收费、周转率和违停 -> 支撑政策评估
场景6：车位主或商圈运营人员 -> 发布共享车位/营销活动 -> 用户预约或核销 -> 查看收益和效果
```

## 二、竞品分析

### 2.1 主要竞品

| 产品 | 优势 | 不足 |
|------|------|------|
| ETCP停车 | 覆盖广、车牌识别成熟、支付体验较好 | 各车场独立运营，城市级监管和统一诱导能力弱 |
| 捷停车 | 商业停车场接入较多，线上缴费成熟 | 依赖合作车场，路内停车、共享停车和政策分析不足 |
| 停简单 | 支付体验流畅，停车记录清晰 | 更偏支付工具，缺少设备运维、动态定价、监管分析 |
| 商场自有停车系统 | 与会员和消费体系结合紧密 | 用户入口碎片化，跨商圈和跨车场能力弱 |

### 2.2 差异化定位

| 维度 | 常见竞品 | 本项目定位 |
|------|:---:|:---:|
| 城市级停车一张图 | 部分车场覆盖 | 汇聚路内车位和路外停车场，统一展示空位、价格、距离 |
| 预约与导航闭环 | 部分支持 | 推荐停车方案、预约锁定15分钟、导航到场、到场确认 |
| 无感支付 | 较成熟 | 实训阶段模拟车牌识别和扣费，保留真实支付扩展接口 |
| 共享停车 | 覆盖较少 | 支持机关、楼宇、小区闲置车位错时共享 |
| 监管与运维 | 较弱 | 提供城市监管平台、设备运维平台、违停取证和政策评估 |
| 商圈营销 | 多为单商场内部能力 | 支持多商户停车优惠、停车券核销和活动效果统计 |

## 三、技术分析

### 3.1 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端 | Vue 3 + Element Plus + ECharts + Vite | 组件化UI、数据可视化、快速本地开发 |
| 后端 | FastAPI + SQLAlchemy | RESTful API、Swagger接口文档、分层业务服务 |
| 数据库 | SQLite | 轻量、零部署，适合课程实训和本地演示 |
| AI视觉 | 模拟检测数据 + 人工复核 | 不训练真实模型，用检测结果表和状态面板呈现 |
| 地图/GIS | 列表、区域筛选、坐标点或静态示意图 | 不依赖外部地图服务，保留后续扩展接口 |
| 支付/硬件 | 模拟支付、模拟设备状态 | 不接入真实扣款和真实设备，使用日志和状态字段演示 |

### 3.2 技术可行性

| 关键点 | 判断 | 说明 |
|------|:---:|------|
| 停车场数据联网接入 | 可行 | 用标准化数据表和模拟API接入空位、价格、同步状态 |
| 车位预约锁定 | 可行 | 使用状态机实现free、locked、occupied、disabled等状态 |
| 无感支付模拟 | 可行 | 车牌入出场、计费、支付结果均可由后端服务模拟 |
| 大数据分析 | 可行 | 基于订单、车位状态、预约记录做聚合统计和ECharts展示 |
| 动态定价 | 可行 | 用规则引擎模拟区域热度、时段、利用率等调价因素 |
| 设备运维和监管 | 可行 | 通过设备台账、告警记录、汇总指标实现演示闭环 |
| 三人并行开发 | 可行 | 按数据运营监管、车主停车路径、车主端扩展场景三条线并行 |

## 四、功能分析

### 4.1 功能全景（18个一级模块）

| 编号 | 一级功能模块 | 用户价值 | 实训实现方式 |
|------|------|------|------|
| M1 | 路内停车位视觉检测 | 获取路内车位空闲、占用、违停状态 | 模拟检测数据、人工复核、状态同步 |
| M2 | 停车场数据联网接入 | 汇聚路外停车场实时空位和价格 | 标准化接入表、模拟同步、离线告警 |
| M3 | 城市停车一张图 | 一站式查看全市停车资源 | 列表/地图模拟、区域筛选、空位热度 |
| M4 | 停车诱导信息发布 | 将车辆引导至合适停车场 | 诱导屏台账、内容生成、发布记录 |
| M5 | 智能导航与车位预约 | 推荐停车方案并锁定车位 | 推荐排序、15分钟锁定、预约状态机 |
| M6 | 车牌识别与无感支付 | 提高入出场和缴费效率 | 车牌模拟识别、计费、支付状态 |
| M7 | 停车行为大数据分析 | 支撑运营优化和政策分析 | 周转率、平均时长、饱和度、热度统计 |
| M8 | 动态定价策略引擎 | 通过价格调节停车供需 | 规则引擎模拟调价、价格公示 |
| M9 | 共享停车管理平台 | 提高闲置车位利用率 | 共享时段发布、预约、防冲突 |
| M10 | 充电车位智能管理 | 支持新能源车停车和充电 | 充电桩状态、预约、异常占用告警 |
| M11 | 违停自动抓拍与取证 | 提升违停监管效率 | 模拟抓拍记录、证据审核、处理状态 |
| M12 | 长期停车月卡管理 | 支持长期停车用户续费管理 | 月卡申请、审批、续费、到期提醒 |
| M13 | 车场运营管理后台 | 提供车场日常运营能力 | 车位、订单、收入、设备、远程操作 |
| M14 | 车主移动端APP/小程序 | 提供车主全流程停车服务 | 响应式车主端、搜索、预约、支付、记录 |
| M15 | 反向寻车导航 | 帮助车主快速找到车辆 | 车牌/车位查询、楼层区域、步行指引 |
| M16 | 设备运维管理平台 | 集中管理停车相关设备 | 设备台账、在线率、故障告警、日志 |
| M17 | 城市停车监管平台 | 支撑城市停车态势监管 | 监管大屏、供需分析、收费统计、政策评估 |
| M18 | 商圈停车联合营销 | 打通商户消费和停车优惠 | 营销活动、停车券、优惠核销 |

### 4.2 优先级分级

| 优先级 | 模块 | 理由 |
|:---:|------|------|
| P0 必须有 | M2、M3、M5、M6、M13、M14 | 构成数据接入、车主找位预约、支付和运营后台核心闭环 |
| P1 重要 | M1、M4、M7、M8、M9、M15、M17 | 增强感知、诱导、分析、共享、寻车和监管价值 |
| P2 最好有 | M10、M11、M12、M16、M18 | 课程周期内可通过模拟数据和状态面板实现扩展演示 |

### 4.3 核心功能用例

#### UC-01：城市停车查询与预约

| 项 | 内容 |
|------|------|
| 前置条件 | 系统已接入停车场和车位数据，车主已登录 |
| 触发 | 用户在车主端输入目的地 |
| 流程 | ①输入目的地 ②展示停车一张图/列表 ③按距离、价格、空位筛选 ④选择停车场和车位 ⑤预约锁定15分钟 ⑥到场确认 |
| 后置条件 | 预约状态更新，车位从空闲变为锁定或占用 |
| 异常 | 空位不足提示重新选择；超时未确认自动释放；重复预约被拦截 |

#### UC-02：车牌识别与无感支付

| 项 | 内容 |
|------|------|
| 前置条件 | 车辆绑定车牌，停车场配置费率 |
| 触发 | 车辆模拟入场或出场 |
| 流程 | ①模拟车牌识别 ②生成入场订单 ③出场计算时长和费用 ④模拟扣款 ⑤订单完成并释放车位 |
| 后置条件 | 订单状态为已完成，支付状态为成功，车位重新可用 |
| 异常 | 车牌识别失败进入人工录入；支付失败保留待支付订单 |

#### UC-03：运营管理与监管分析

| 项 | 内容 |
|------|------|
| 前置条件 | 管理员或监管用户拥有相应权限 |
| 触发 | 用户进入运营后台或监管平台 |
| 流程 | ①查看车位和订单概览 ②查看收入、周转率、饱和度 ③处理设备/违停异常 ④导出统计报表 |
| 后置条件 | 操作记录进入日志，统计指标更新 |
| 异常 | 权限不足禁止访问；设备离线进入告警；监管数据需脱敏展示 |

## 五、可行性分析

### 5.1 技术可行性：高

- 核心业务以CRUD、状态机、统计聚合和规则计算为主，适合Vue3 + FastAPI + SQLite实现。
- AI视觉、GIS地图、真实支付、真实硬件接入不作为实训强依赖，使用模拟数据和预留接口呈现。
- 前后端分离后，三名成员可以按模块并行开发和联调。

### 5.2 经济可行性：高

- 本地开发运行，不需要购买服务器、硬件设备、支付接口或地图服务。
- 使用开源技术栈和模拟数据，课程演示成本可控。

### 5.3 时间可行性：可控

| 阶段 | 时间 | 产出 |
|------|------|------|
| 第1-2周 | 5.25-6.7 | 立项、项目计划、需求分析、18模块口径确认 |
| 第3周 | 6.8-6.14 | 概要设计、数据库设计、接口设计、页面设计 |
| 第4-6周 | 6.15-7.5 | 核心闭环开发、扩展模块模拟实现、前后端联调 |
| 第7-8周 | 7.6-7.18 | 测试修复、文档完善、验收演示 |

### 5.4 可行性结论

项目具备实施可行性。核心闭环聚焦停车场接入、一张图查询、预约、车牌识别计费、车主端和运营后台；扩展能力通过模拟数据、状态面板和接口预留完成，既贴合项目要求，又能在课程周期内稳定演示。

## 六、非功能性需求

| 类型 | 要求 |
|------|------|
| 性能 | 页面首次加载不超过2秒，简单查询API响应不超过500ms，复杂统计不超过1秒 |

| 可扩展性 | AI识别、地图、支付和硬件设备均保留扩展接口，可由模拟服务替换为真实服务 |
