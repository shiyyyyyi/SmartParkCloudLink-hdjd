## 四、技术方案（本周新增）

### HLD设计要点：
- **系统架构**：前后端分离B/S架构，Vue3 SPA + FastAPI RESTful + SQLite
- **模块设计**：18个一级模块按P0/P1/P2三级优先级分层设计
- **数据库设计**：核心数据表（含共享车位、设备、诱导屏、月卡、营销等）+ ER关系 + 索引设计
- **接口设计**：40+ RESTful API端点，统一返回格式`{code, msg, data}`
- **前端设计**：Vue Router路由表 + 组件树 + Pinia状态管理
- **后端设计**：FastAPI目录结构 + 业务服务层 + 认证中间件
- **模板化交付**：以交付物模板为母版生成SRS/HLD提交文档，保留模板封面、目录层级和正文格式，仅替换并扩展项目内容

### SRS设计要点：
- 18个一级功能模块完整需求规格（含AI视觉、GIS地图、无感支付、硬件接入模拟实现策略）
- 每模块含功能点拆分（4-6个F点）、业务规则（BR）、异常处理（EX）、验收标准
- 需求追踪矩阵（RTM）覆盖全部18个一级模块
- 8项评审问题全部修订闭合

## 五、遇到的问题与解决

| 问题 | 解决方案 |
|------|------|
| 课题原始18功能模块部分超出实训可实现范围（AI视觉、GIS、真实支付、硬件接入） | 采用"需求完整定义+模拟数据/接口预留实现"策略，核心闭环优先完成，高难度外部能力模拟呈现 |
| 需求评审中发现8项遗漏/矛盾/不明确项 | 逐项修订闭合，补充缺失功能点，明确模糊定义 |
| 18模块与3人团队的工作量平衡 | 按责任域分工：程晓洋负责数据运营监管，程子浩负责车主停车路径，丁梓钊负责车主端与扩展场景 |
| 原始HLD模板为.doc格式，直接编辑不利于后续自动化检查 | 使用Word转换为.docx母版后再生成提交版，模板目录未修改；完成后导出PDF用于版式复核 |
