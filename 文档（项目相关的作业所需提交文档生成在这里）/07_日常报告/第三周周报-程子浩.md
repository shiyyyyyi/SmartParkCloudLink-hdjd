## 三、RA需求细化成果

### 细化策略
1. **逐模块拆解**：将课题18个功能模块的简要描述拆解为：
   - 功能点（F1-Fn）：每个模块4-6个可验证的功能点
   - 输入/输出：明确定义每个功能点的数据流入流出
   - 业务规则（BR）：约束条件和计算逻辑
   - 异常处理（EX）：异常场景和处理方式
   - 验收标准：可测试的Checklist

2. **模拟边界处理**：AI视觉、GIS地图、真实支付和硬件接入因实训环境限制无法完整接入，采用"需求完整定义+模拟数据/接口预留实现"策略

3. **状态机设计**：为M5预约模块设计了完整的状态机模型

### 关键细化项
- M5预约状态机：free ↔ locked → occupied → free
- M6计费公式：费用 = 停车时长(小时) × 费率，支持免费时长
- 高难度能力模拟策略：需求规格完整定义，编码阶段使用状态面板、模拟数据和预留接口展示

## 五、遇到的问题与解决

| 问题 | 解决方案 |
|------|------|
| 课题需求描述模糊（如"实时""智能"等定性词） | 逐一量化：实时=30秒刷新、智能推荐=多条件排序 |
| 18个模块中部分超出实训可实现范围 | 采用分层策略：核心闭环完整实现，高难度外部能力用模拟数据和接口预留实现 |
| 需求评审中角色视角不同导致意见分歧 | 以"可设计、可测试、可追溯"为统一评判标准 |
| SRS扩写后需要确认个人负责模块与HLD设计是否同步 | 按M3/M4/M5/M6/M15/M18逐项复核功能点、接口、数据库表和异常处理，保证第4周编码可直接落地 |
| HLD第5章需要覆盖后端开发中的异常分支 | 从RA/QA视角检查预约冲突、预约超时、支付失败、车牌识别失败和设备离线等场景，确认均已有错误码和处理流程 |
